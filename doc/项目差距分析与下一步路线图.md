# Skills Hub 当前实现差距分析与下一步路线图

## 一、文档目的

本文件用于说明当前仓库实现与《Skill Server 技术方案 v1》之间的主要差距，并给出后续迭代优先级，作为文档校准和后续开发排期的依据。

---

## 二、总体判断

当前项目已经具备：

- 后端 FastAPI 主服务
- 前端 Vue3 主页面
- Skill 管理、Git 同步、版本、搜索、时间线等核心模块
- 本地开发和 Docker 部署能力

但项目现阶段更适合定义为：

> **具备核心主链路的 MVP / Beta 版本，而非技术方案中的完整目标态。**

---

## 三、主要差距清单

### 3.1 P0：必须优先处理

#### 1. 只读匿名模式没有真正闭环

现状：

- 运行模式配置存在
- 前端大多数页面仍按“需要登录”设计
- 后端多个关键接口仍依赖必须登录用户

影响：

- 技术方案中“登录开启 / 只读匿名 / 初始化管理”三种模式并未真正全部可用

目标：

- 允许匿名访问搜索、详情、时间线、下载
- 拒绝匿名执行创建、编辑、删除、同步、收藏、通知等写操作

#### 2. Git Skill 的详情和文件读取链路未完全统一

现状：

- Git Skill 已可导入
- 但部分详情、文件读取逻辑仍默认按 private skill 路径处理

影响：

- Git 来源 Skill 的查看体验不稳定
- “private / git 两类 Skill 统一管理”这一目标未完全达成

目标：

- 所有详情、预览、下载、文件读取统一以 `skill.storage_path` 为准

#### 3. 项目文档状态失真

现状：

- 原《项目完整度报告》将项目描述为 100% 完成

影响：

- 对内外判断偏差
- 容易误导后续排期和验收

目标：

- 统一将项目状态校准为“核心 MVP 已完成，仍有关键闭环待补”

---

### 3.2 P1：应该尽快完成

#### 4. APScheduler 定时同步未落地

现状：

- 依赖已在后端声明
- 配置项已存在
- 应用启动未接入调度器

目标：

- 在应用生命周期中初始化调度器
- 定期触发自动同步的 Git 仓库

#### 5. 资源角色只有定义，没有管理闭环

现状：

- owner / maintainer / editor / viewer 权限矩阵已定义
- 创建资源时会自动赋予 owner
- 缺少完整的角色分配、撤销、转让能力

目标：

- 提供资源角色管理 API
- 提供前端角色管理入口
- 与审计日志联动

#### 6. 审计日志和通知没有形成业务闭环

现状：

- 数据模型和服务已经存在
- 但关键事件未系统性写入

目标：

- 对创建、删除、同步、权限变更、拒绝访问等操作写入审计日志
- 对需要提醒的业务事件触发通知

---

### 3.3 P2：中期补齐

#### 7. OIDC 尚未实现

现状：

- 当前只有配置项，没有实际登录流程

目标：

- 增加 OIDC 登录入口、回调处理、用户映射与前端配套

#### 8. 文件预览安全闭环不足

现状：

- 已有 PreviewService
- 主流程仍大量依赖直接读取和前端渲染

目标：

- 统一使用安全预览链路
- 对 HTML / SVG 等高风险类型使用更严格策略

#### 9. 搜索业务统计未完全闭环

现状：

- 搜索排序依赖 favorite_count / download_count / usage_count
- 只有部分统计有完整写回逻辑

目标：

- 补齐下载与使用统计
- 让 blend 排序基于真实数据工作

---

## 四、建议的实施顺序

### 阶段一：修正基础闭环

1. 打通只读匿名模式
2. 修复 Git Skill 详情 / 文件读取链路
3. 补齐相关测试
4. 同步修正文档说明

### 阶段二：补齐平台治理能力

1. 接入 APScheduler 自动同步
2. 完成资源角色管理
3. 落地审计日志写入与查询
4. 加强通知联动

### 阶段三：补齐企业化能力

1. 落地 OIDC
2. 补强预览安全
3. 完善系统配置管理与运维能力

---

> Skills Hub 已完成核心 MVP，主干业务流程已具备，当前正处于从可演示版本向完整产品化能力补齐的阶段。
